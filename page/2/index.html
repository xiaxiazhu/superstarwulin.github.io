  • 具体配置方法

mac上配置opencv,xcode c++ commond line 使用

OpenCV 2.4.6 + mac Xcode 4.6 安装教学

前一阵子刚换mac，於是打算用mac来跑opencv，但没想到安装起来还满复杂的，

於是就决定做一个安装教学，希望可以帮助想在mac上跑opencv的各位

安装环境：

mac lion mountain 10.8

opencv 2.4.6

Xcode 4.6

1.下载opencv：

右边有一个opencv for linux/mac

点下去就会自动下载了

2.安装homebrew：

这是一个很好用的mac安装许多程式的软体

只要在终端机上输入

ruby -e “$(curl -fsSL https://raw.github.com/Homebrew/homebrew/go/install)”

就会自动下载安装了

终端机可以在下图找到

3.安装cmake：

在终端机输入

brew install cmake

就会自动下载安装了

4.安装Xcode：

直接去App Store 下载安装，Xcode是免费的

安装完后 进入Xcode的 preference 里面的download

下载 command line tool

5.安装opencv：

将下载回来的opencv压缩档解压缩会跑出资料夹

使用终端机进入解压缩后的资料夹

比较不熟linux指令的人

可以先用在终端机输入 => ls ,看现在位置的资料夹

再终端机依序输入 => cd 资料夹名称 ,进入压缩的资料夹中

可以在终端机输入 => pwd ,确认当前位置

接着依序输入

mkdir release

cd release

cmake -G “Unix Makefiles” ..

make

sudo make install

依序完成以上步骤就全都安装好啰

接下来是新建opencv project

6.建立opencv project：

开启Xcode 建立专案

建立名称后 下面type选择 c++

建立后到build setting中

build options -> compiler for c/c++…

选择LLVM GCC 4.2

到search paths中设定路径

library search paths 中 输入 /usr/local/lib

header search paths 中输入 /usr/local/include /usr/local/include/opencv

7.include lib：

在专案名称上面按右键

选择add files to…..

进入后直接按下 / 就会跳出go to the folder 再输入 /usr/local/lib

接着把所有的opencv2.4.6档案都加进去（因为我以前有安装过2.4.3 所以有2.4.3的文件）

以上所有环境都设定好啰

可以开始试试看程式了！！！

8.final test：

贴上以下code

//  Created by vince on 2013.
//  Copyright (c) 2013年 vince. All rights reserved.

#include "opencv2/core/core.hpp"
#include "opencv2/highgui/highgui.hpp"
#include "opencv2/imgproc/imgproc.hpp"

using namespace cv;
using namespace std;

int main(int argc, char *argv[])
{
    cv::Mat src;

    src=cv::imread("/Users/mac/Documents/MyCode/biglena.bmp", -1);

    //上面的路径为图片路径

    cv::imshow("show image",src);  //这边会show图出来

    cv::waitKey(0);

    return 0;
}

以上如果看得到图片就大功告成啰～～

浅谈前端的主流设计模式

基本概念

设计模式之所有是个计算机学科里面各类语言一直以来研究的问题，是因为他和架构相关，解决亘古不变的问题 – 当代码规模大到一定程度是否可以符合业务变更、性能、可持续开发的要求。代码规模变大，解决方案变得复杂，随之出现细分，以及细分之后的管理关系，这些管理关系就是我们要聊的。

前端的设计模式

前端通常情况下是在一定的逻辑下展现相应的UI，UI和逻辑的解耦可以提高迭代的效率，比如：模板与前端脚本分开来写让改UI和改逻辑更加清晰化。然而谁也没办法解决一个逻辑和UI分化之后的一个哲学问题– UI和逻辑是一起的，但他们却不是一起的。

软件从处理一件事物发展到处理许多事物，各各事物之间有包含、主次、顺序等等的关系，变得越来越浮渣。因为数据和逻辑庞大了，所以数据与逻辑就分离了，然后事件和业务分离。如果还要细分，当然还会有更多的说法。

当一个问题出现一次的时候它是一个问题，当一个问题出现无数次它会成为历史，当一个问题将会出现无数次的时候，它将需要一个明确的定义和解决方案。那么，数据的更新和界面的更新这一问题本放大了无数次呗，因为它就是前端需要解决的主要问题。那我们来看下这一主要问题前端工程里面出现的关系。

Model View Control

最原始的模型架构细分管理模型，MVC，相互关系如上图：推动业务的事件由用户与view交互，通过view向controller发动，然后触发数据更新。而mvc的使用发现的问题是当有变化发生时候，需要维护三个对象和三个三个交互，这让事情复杂化了。

mvc的三角关系被认为给一些人带来了麻烦，所以需要催生更好的解决方案。

Model View Presenter

Model View ViewModel

ViewModel大致上就是MVP的Presenter和MVC的Controller了，而View和ViewModel间没有了MVP的界面接口，而是直接交互，用数据“绑定”的形式让数据更新的事件不需要开发人员手动去编写特殊用例，而是自动地双向同步。数据绑定你可以认为是Observer模式或者是Publish/Subscribe模式，原理都是为了用一种统一的集中的方式实现频繁需要被实现的数据更新问题。
